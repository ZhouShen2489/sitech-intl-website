# BOSS / BSS-OSS 运营支撑

## 页面定位

BOSS 页面要讲清楚：思特奇不是只做一个 CRM 或计费模块，而是能把运营商客户、产品、订单、开通、计费、结算和运营管控接成一条生产运营链路。海外表达建议使用 `BSS/OSS operations`、`order-to-billing loop`、`operator-grade operating system` 等买家更容易理解的语言。

## 目标客户

- 基础运营商、区域运营商、复杂通信服务团队。
- 正在做 4G/5G、固话、宽带、流量、云、专线和政企业务融合办理的团队。
- 面临订单、开通、计费、结算跨系统断点的运营团队。

## PPT / 公司主张

- BOSS 是业务运营支撑系统，提供 CRM、计费、开通、结算等前端业务生产和后端服务运营能力。
- 支持客户与产品运营域、网络与服务交互域、合作伙伴管理域、计费账务域、企业管理域、通用能力组件、大数据管理和业务运营管理中心。
- 已支持中国移动、中国联通、中国电信、中国广电等四大运营商数十家省公司。

## 公开市场背景

- TM Forum Open Digital Architecture 强调运营商需要更模块化、可集成、云原生的 IT 架构。
- TM Forum Open APIs 提供产品目录、产品订购、客户管理、计费等电信 API 方向，适合用来解释 BSS/OSS 从封闭系统走向可组合能力。
- 参考来源：
  - https://www.tmforum.org/open-digital-architecture/
  - https://www.tmforum.org/open-digital-architecture/about-open-apis/

## 网站内容角度

- 不要写成“大而全系统介绍”，要写成客户能理解的闭环：客户和产品 -> 订单和开通 -> 计费和结算 -> 运营管控。
- 第一屏强调“让运营商业务从办理到计费真正跑成闭环”。
- 关键 CTA：订单闭环诊断、计费结算规则梳理、政企业务或新业务支撑试点。

## 推荐页面模块

- Hero：BSS/OSS 运营支撑，一句话讲清订单到计费闭环。
- Buyer pain：业务上线慢、订单断点多、计费结算压力大。
- Capability map：客户与产品、订单与服务交互、计费账务与伙伴。
- Proof notes：四大运营商省公司经验，注意标注为 PPT/company claim。
- First scope：选择一个订单到计费链路先验证。
